\chapter*{Introducción}
\section*{Descripción}

Este proyecto consiste en la implementación de un intérprete de un subconjunto del lenguaje funcional ML, que devolverá como salida el tipo de las expresiones ingresadas. Para esto se utilizaron las herramientas automáticas JLex y CUP aplicadas en el entorno del lenguaje de programación Java, que se encargan de generar un parser ascendente que reconoce las cadenas ingresadas y aplica las reglas semánticas convenientemente definidas.

\section*{Archivos entregados}
El proyecto está estructurado en tres carpetas:
\begin{itemize}
	\item /Casos de Test: incluye diversos archivos de test (válidos, inválidos, y con bugs).
	\item /doc: incluye los pdf con la documentación del proyecto.
	\item /src: incluye todos los archivos necesarios para compilar el proyecto:
		\begin{itemize}
			\item /JLex: carpeta que contiene todos los archivos propios de la herramienta JLex.
			\item /java\_cup: carpeta que contiene todos los archivos propios de la herramienta CUP.
			\item Attrs.java: clase auxiliar definida para guardar los datos de los no-terminales en la gramática; a grandes rasgos, para cada no-terminal se recuerda el tipo de la expresión que sintetiza, y una lista de \emph{related-ids}; cada identificador que apunta a este objeto en la tabla de símbolos es indicado a su vez en esta lista, lo que permite, por ejemplo, detectar cuándo dos identificadores son el mismo tipo de politipo, o cambiar el tipo de un grupo de identificadores de "`politipo"' a "`real"'.
			\item lex.lex: archivo que define los patrones de los tokens devueltos por el analizador léxico generado automáticamente con la herramienta JLex.
			\item cup.cup: archivo que define la gramática y las reglas semánticas utilizadas por el parser generado automáticamente con la herramienta CUP.
			\item bat.bat: script de consola para Windows que se encarga de eliminar archivos anteriores, generar el analizador léxico, el parser, y recompilar el intérprete para ser ejecutado desde consola.
			\item sh.sh: script de consola para sistemas Linux/Unix, que realiza las mismas funciones que el bat.
		\end{itemize}
\end{itemize}

\chapter{Modo de uso}
\$ java MLOR \emph{[inputFile]},

donde \emph{inputFile} es un archivo con código ml que puede ingresarse opcionalmente.  Si bien el programa no fue pensado específicamente para ser utilizado con archivos de entrada, se incluyó la opción para facilitar la corrección del intérprete por parte de la cátedra. Si efectivamente se especifica un archivo, para cada una de las expresiones que contenga, el intérprete mostrará el tipo como la cátedra solicitó. En caso de que {\bf no} se especifique ningún archivo, el intérprete mostrará un prompt para leer expresiones y funciones por pantalla, y mostrará el tipo de cada una de ellas. MLOR está pensado para ser utilizado con el prompt, por lo que al ingresar los casos de prueba directamente desde un archivo el formato de salida puede no ser el ideal.

Para realizar una asignación se utiliza val \emph{X} = \emph{Y};, siendo \emph{X} el nombre de la variable que recibirá el resultado, e \emph{Y} la expresión que se asignará a \emph{X}. En caso de que únicamente se ingrese una expresión, ésta se ligará por defecto a la variable \emph{it}.

Ejemplos:
\begin{lstlisting}[frame=single]
MLOR> val x = 5;
val x: int;
\end{lstlisting}
En este caso, el valor entero 5 se liga a la variable global \emph{x}.


\begin{lstlisting}
MLOR> 11.11
val it: real;
\end{lstlisting}
En este caso, el real mostrado se liga implícitamente a la variable global \emph{it}.

Para declarar una función, se ingresa el prefijo \emph{fun} en el intérprete seguido de los parámetros correspondientes y su expresión:

\begin{lstlisting}
MLOR> fun f x = x + 5;
val f: int -> int = fn;
MLOR> fun g x y = (f x > 10) orelse (x>y);
val g: int -> int -> boolean = fn;
\end{lstlisting}
En este ejemplo se declaran dos funciones: \emph{f}, con parámetro \emph{x}, y \emph{g}, con parámetros \emph{x} e \emph{y}, que a su vez utiliza la función f.

Tanto las variables globales como los identificadores de las funciones son recordados internamente en el parser en una tabla de símbolos (implementada con una tabla hash de Java). En dicha tabla se mantiene como llave el identificador, y la entrada de la tabla contiene un objeto que es instancia de la clase Attrs; a medida que los identificadores se relacionan en la expresión, pueden apuntar a la misma instancia de Attrs; por ejemplo, en una expresión como la del último ejemplo, \emph{x} e \emph{y} apuntarán a la misma instancia de Attrs. Se detalla con mayor profundidad cómo se decide el tipo de las expresiones en la sección {\bf Analizador Semántico}.

\section{Requerimientos}

Dada la tecnología utilizada para realizar el intérprete, será necesario contar con la máquina virtual de Java (oficialmente fue probado para la versión 1.6) para poder correr el archivo ejecutable \emph{MLOR}. En caso de desear compilar manualmente el intérprete en Windows, las variables \emph{class} y \emph{classpath} deben estar correctamente seteadas.

\section{Formato de salida}

El formato de salida utilizado es igual al del lenguaje original ML: todas las expresiones poseen un tipo determinado, y todas las funciones poseen un dominio y un rango, que se representan mediante la notación Dominio -\textgreater Rango. Las expresiones ingresadas en el intérprete y pueden ser de tipo \emph{int}, \emph{real}, \emph{boolean}, \emph{string}, \emph{list} (que a su vez tiene un tipo asociado) y \emph{tuple} (que tiene un tipo asociado por cada uno de sus elementos). En el caso particular de las funciones, se consideran dos tipos especiales además de los mencionados: el politipo, y el politipo con igualdad. Dado que puede haber muchas instancias de politipos diferentes, los primeros se notan con un apóstrofe seguido del identificador del politipo, mientras que los segundos son precedidos por \emph{dos} apóstrofes.


\begin{lstlisting}
MLOR> 5;
val it: int
\end{lstlisting}
Ejemplo de una expresión entera ligada implícitamente a \emph{it}.


\begin{lstlisting}
MLOR> val x = 5;
val x: int;
\end{lstlisting}
La misma expresión entera, ligada explícitamente a \emph{x}.


\begin{lstlisting}
MLOR> val y = (1,2.0);
val y: (int * real);
\end{lstlisting}
Una tupla de tipo (int * real) ligada explícitamente a \emph{y}.


\begin{lstlisting}
MLOR> fun f x = 5;
val f: 'a -> int = fn;
\end{lstlisting}
Una función f que recibe un parámetro de cualquier tipo y  devuelve el número entero 5.

\begin{lstlisting}
MLOR> fun g x y = x=y;
val f: ''a -> ''a -> boolean = fn;
\end{lstlisting}
Una función g que recibe dos parámetros del mismo tipo \emph{que soporte igualdad}, los compara, e indica si son iguales o no.

\begin{lstlisting}
MLOR> fun h x y = (x,y);
val h: 'a -> 'b -> ('a * 'b) = fn;
\end{lstlisting}
Una función h que recibe dos parámetros de cualquier tipo \emph{(no necesariamente igual)}, y devuelve una tupla con estos dos elementos.


\chapter{Analizador léxico}
\section{Tokens}
El Analizador Léxico de MLOR es generado gracias a la utilización de la herramienta automática JLex. Dicho analizador reconoce los tokens especificados en la tabla \ref{tab:tokens}, y los pasa \emph{on demand} al parser.

\section{Palabras reservadas}
El analizador léxico de MLOR mantiene algunas palabras reservadas que no puede ser redefinidas por el usuario para nombrar una función o un valor; dichas palabras son los operadores lógicos \emph{orelse} y \emph{andalso}, los operadores aritméticos \emph{div} y \emph{mod}, el operador unario \emph{not}, el indicador de lista vacía \emph{nil}, el prefijo de las funciones \emph{fun}, y el de los valores, \emph{val}.

Nótese que \emph{true} y \emph{false} no son palabras reservadas, sino variables booleanas insertadas por defecto en la tabla de símbolos, por lo que podrían redefinirse al valor que el usuario del intérprete desee.

Es interesante destacar que, a diferencia del lenguaje original ML, el analizador léxico de MLOR {\bf no} es \emph{case-sensitive}. Si bien las expresiones regulares de las palabras reservadas mostradas en la tabla \ref{tab:tokens} están desarrolladas de manera exhaustiva, la propiedad para que el lexer no sea \emph{case-sensitive} es alcanzada gracias al setting \texttt{\%ignorecase} de la herramienta JLex.

La expresión regular de la entrada de "`String"' en la tabla significa "`cualquier conjunto de caracteres sin considerar `` y el salto de línea \textbackslash n"', rodeado de comillas dobles.

\chapter{Analizador Sintáctico y Semántico}
Una vez superados los checkeos del Analizador Léxico, el control del parser pasa a estar a cargo del Analizador Sintáctico, que dado un grupo de tokens provistos por el Lexer se asegura de que éstos formen un grupo de sentencias coherentes, o reporta el error encontrado. De acuerdo a la regla de derivación utilizada, se aplican reglas semánticas para que la evaluación de cada una de las sentencias resulte en lo esperado. A continuación detallaremos qué es lo que espera recibir el Analizador Sintáctico, el tipo de errores reportados en caso de no encontrarlo, y el alcance en el reconocimiento semántico de las expresiones en el parser.

\section{Alfabeto de entrada}

La gramática original fue ligeramente modificada para eliminar algunas ambiguedades (por ejemplo, el hecho de que una derivación de una tupla de un único elemento y una expresión parentizada podían resultar iguales). Además, los operadores de las expresiones fueron separados utilizando las reglas de precedencia definidas en ML (y mencionadas adecuadamente en el enunciado entregado para la realización del intérprete). La gramática resultante, utilizada por CUP para la generación del parser, es:

\begin{verbatim}

<Programa> ::= <Lista_sentencias> PUNTO

<Lista_sentencias> ::= <Lista_sentencias> <Sentencia> PTOCOMA | <Sentencia> PTOCOMA

<Sentencia> ::= VAL ID = <Exp_Rel> | <Exp_Rel> | FUN <MatchingFunc>	

<MatchingFunc> ::= ID ID <CurryingList> | ID <TuplePattern> IGUAL <Exp_Rel>

<CurryingList> ::= ID <CurryingList> | IGUAL <Exp_Rel>

<TuplePattern> ::= PARABRE <TuplePattern2> PARCIERRA

<TuplePattern2> ::= <TuplePattern2> COMA ID  | ID

<Exp_Rel>  	::= 	<Exp_Rel> OPREL <Exp_cons> | <Exp_Rel> DISTINTO <Exp_cons> | <Exp_Rel> IGUAL <Exp_cons> | <Exp_cons>

<Exp_cons> 	::=	<Exp_add> CONS <Exp_cons>| <Exp_add>

<Exp_add>  	::=	<Exp_add> OPADD <Exp_mul> | <Exp_add> ORELSE <Exp_mul> | <Exp_add> CONCAT <Exp_mul> | <Exp_mul>

<Exp_mul>  	::=	<Exp_mul> MUL <Exp_un> | <Exp_mul> DIVREAL <Exp_un> | <Exp_mul> DIVINT <Exp_un> | <Exp_mul> MOD <Exp_un> | <Exp_mul> ANDALSO <Exp_un> | <Exp_un>

<Exp_un>   	::=	NOT <Exp_fun> | MINUS<Exp_fun> | <Exp_fun>

<Exp_fun>  	::= 	<Exp_fun> <Exp_atom> | <Exp_atom>

<Exp_atom>	::=	INT | REAL | STRING | ID | NIL | PARABRE <Exp_Rel> PARCIERRA | <Tuple> | <List>

<Tuple> 	::= PARABRE <Exp_Rel> COMA <Tuple2> PARCIERRA

<Tuple2> 	::= <Tuple2> COMA <Exp_Rel> | <Exp_Rel>

<List> 		::= CORABRE <List2> CORCIERRA

<List2> 	::= <List2> COMA <Exp_Rel> | <Exp_Rel>
\end{verbatim}
\section{Errores detectados}
MLOR posee un manejo de errores básico en la interpretación de código; los tipos de errores detectados son:

\begin{itemize}
	\item \textbf{Lexical Error:} ocurre cuando se intenta ingresar un caracter que no pertenece al lenguaje de entrada.
		\begin{lstlisting}
		MLOR> $;
		Error: Lexical Error: $ at line 1
		\end{lstlisting}
	\item \textbf{Undeclared identifier:} se presenta al intentar evaluar el valor de una variable que no se encuentra ligada.
		\begin{lstlisting}
		MLOR> val x = y+2;
		Error: Undeclared identifier (y)
		\end{lstlisting}
	\item \textbf{Incompatible types:} ocurre cuando se utilizan dos tipos \emph{diferentes} en operadores que esperan tipos específicos.
		\begin{lstlisting}
		MLOR> val x = 1 + 2.5;
		Error: Incompatible types.
		\end{lstlisting}
	\item \textbf{Wrong argument type:} se presenta al utilizar tipos de argumentos inválidos con determinados operadores.
	\begin{lstlisting}
	MLOR> val x = "string1" + "string2";
	Error: Integer or real expected.
	\end{lstlisting}
	\begin{lstlisting}
	MLOR> val x = ~"string";
	Error: ~ operator can only be applied to integer and real values.
	\end{lstlisting}
	\item \textbf{List of different types:} aparece cuando se construye una lista que tiene al menos dos tipos diferentes de elementos.
	\begin{lstlisting}
	MLOR> val x = ["a", 3];
	Error: The list has elements of different types.
	\end{lstlisting}
	\item \textbf{Otros:} errores inesperados para el parser, como por ejemplo, declarar una función sin parámetros:
	\begin{lstlisting}
	MLOR> fun f = 1;
	Syntax error
	Couldn't repair and continue parse
	Error: Can't recover from previous error(s)
	\end{lstlisting}
\end{itemize}

\section{Alcance semántico del parser}
MLOR provee el reconocimiento de diversos tipos de sentencias y combinaciones de constructores; se detallarán en esta sección los casos y combinaciones más importantes; se adjuntan además en la carpeta \texttt{/bin} diversos casos de testing para probar su funcionamiento;

Por empezar, MLOR permite la ligadura explícita del resultado de sentencias a variables utilizando el prefijo \emph{val}. En caso de no especificar una variable destino, el resultado de la sentencia ingresada será ligado implícitamente a la variable global \emph{it}. Dado que se utiliza una tabla de símbolos para recordar los atributos de cada variable, y dicha tabla persiste durante la ejecución del parser, las variables podrán ser utilizadas en sucesivas sentencias.

\begin{lstlisting}
MLOR> "Hello world!";
val it: string
MLOR> val myString = it;
val myString: string
\end{lstlisting}

MLOR también permite ingresar listas de cualquier tipo básico:

\begin{lstlisting}
MLOR> val myStringList = ["Hello", "World!"];
val myStringList: string list
MLOR> val myIntList = [4, 8, 15, 16, 23, 42];
val myIntList: int list
\end{lstlisting}

Tuplas de cualquier tipo:

\begin{lstlisting}
MLOR> val myTuple = (1, 2.0, "Three", [4]);
val myTuple: (int * real * string * int list)
\end{lstlisting}

Y combinaciones de todas ellas:

\begin{lstlisting}
MLOR> val myComplicatedList = [(1, 1.0, "First item on the list", [1]), (2, 2.0, "Second item", [2]), (3, 3.33, "Yet another item", [33])];
val myComplicatedList: (int * real * string * int list) list
MLOR> val aHardTuple = ([1.5, 2.0, 3.0], "Yepe, it seems hard.", 17, 2035);
val aHardTuple: (real list * string * int * int)
\end{lstlisting}

Además de la ligadura de variables, MLOR permite declarar funciones, que deberán poseer un identificador y al menos un parámetro:

	\begin{lstlisting}
	MLOR> fun f x = 5;
	val f: 'a -> int = fn;
	\end{lstlisting}
	
Las funciones también serán recordadas en la tabla de símbolos, por lo que podrán ser utilizadas posteriormente en otras funciones y expresiones:
	
	\begin{lstlisting}
	MLOR> val g = f 3;
	val g: int
	\end{lstlisting}
	
MLOR cambia el tipo dinámicamente y liga las variables correspondientes en cualquier tipo de expresión; para las operaciones aritméticas, el valor por defecto es entero:

	\begin{lstlisting}
	MLOR> fun defaultIntAddition x y = x + y;
	val defaultIntAddition: int -> int -> int = fn;
	\end{lstlisting}

Sin embargo, en caso de ser necesario, y siempre y cuando esté permitido por ML, las variables pueden cambiar de tipo:
	
	\begin{lstlisting}
	MLOR> fun actualRealAddition x y = x + y + 0.0;
	val actualRealAddition: real -> real -> real = fn;
	\end{lstlisting}
	
Este cambio de tipo (válido, y permitido por ML) puede ocurrir con cualquier tipo de expresiones:

	\begin{lstlisting}
	MLOR> fun tupleFunction (x,y) = (x > (y*13.37));
	val tupleFunction: (real * real) -> boolean = fn;
	\end{lstlisting}
	
Y en cualquier tipo de funciones:

	\begin{lstlisting}
	MLOR> fun f x y z = (x 3.0, z y [2],y nil);
	val f: (real -> 'a) -> ('b list -> 'c) -> (('b list -> 'c) -> int list -> 'd) -> ('a * 'd * 'c) = fn;	
	MLOR> fun productoCartesiano x y = ((x,x), (x,y), (y,x), (y,y));
	val productoCartesiano: 'a -> 'b -> (('a * 'a) * ('a * 'b) * ('b * 'a) * ('b * 'b)) = fn;
	MLOR> fun productoParcial unElem = productoCartesiano unElem;
	val productoParcial: 'a -> 'b -> (('a * 'a) * ('a * 'b) * ('b * 'a) * ('b * 'b)) = fn;	
	\end{lstlisting}
	
	
\chapter{Definición Dirigida por la Sintáxis}
La Definición Dirigida por la Sintáxis se encuentra en el archivo ``dds.pdf''. En ella se encuentra un conjunto de reglas semánticas similares a las utilizadas en el código fuente del intérprete, pero sin los detalles de hilado fino de la implementación para hacerla más legible.

\section{Estructuras utilizadas}
\begin{itemize}
	\item \textbf{\texttt{Attrs}}: El conjunto de atributos de los no terminales. Mantiene el tipo de la subexpresión computada, y la lista de identificadores que referencian a dicho tipo guardados en un conjunto de \emph{related-ids} (implementado internamente a partir de un \texttt{HashSet}). Además, puede contener todos los datos necesarios para los diferentes tipos que puede tener una variable (tipo de elemento para las listas, tipo de dominio y rango para las funciones, etc).
	\item \textbf{Tablas de símbolos}: Se utilizaron dos tablas de símbolos implementadas con \texttt{Hashtables}, con cadenas de caracteres que contienen el identificador como llave, y objetos de tipo Attrs como contenido. La primer tabla es la global, que es utilizada durante toda la ejecución. La segunda es creada cuando se parsean funciones, se utiliza como tabla de símbolos local, y es descartada posteriormente.
\end{itemize}

\subsection{Lista de identificadores}
La lista de identificadores de la clase \texttt{Attrs} tiene una notación particular para los casos en las que una variable la apunta indirectamente.

Por ejemplo, si una variable x es una función con dominio 'a y rango 'b (es decir, su tipo es \texttt{('a -> 'b)}), tanto el objeto \texttt{Attrs} 'a como el 'b deberán saber que están siendo apuntados indirectamente por x. Para esto, en la lista de identificadores se agrega la variable antecedida por un prefijo, que indica por quién está siendo apuntado el objeto. En el caso de los dominios de una función, el prefijo es ``-'', mientras que para el rango es ``\textgreater'', por lo que en las listas contendrán ``-x'' y ``\textgreater x'' respectivamente. Esta información es utilizada luego para corregir las referencias cuando el tipo es cambiado.

Para las listas, el prefijo utilizado es ``['', y para las tuplas ``(i)'', donde \textit{i} es el índice de la referencia en la lista de tipos.

\section{Métodos}
\begin{itemize}
	\item \textbf{tipoMasEspecifico}: Devuelve el tipo más específico entre los dos parámetros, si es que son compatibles. En el caso de las listas, tuplas y funciones, es utilizado recursivamente para comparar que ambos parámetros sean del mismo tipo estructurado, y que los tipos base de éstos también coincidan. Este método también corrige las referencias de las variables en la tabla de símbolos, directa e indirectamente.
	
	En el caso de que sólo uno de los parámetros sea un politipo, se selecciona como tipo más específico el otro, y las entradas de toda la lista de \emph{related-ids} pasan a apuntar al objeto Attrs del tipo más específico en la tabla de símbolos. En el caso de que ambos sean politipos, continúan siéndolo, pero las variables que los apuntaban ahora apuntarán al mismo objeto, y las listas de \emph{related-ids} serán fusionadas.
	
	Por último, si los tipos no son compatibles, el procedimiento frena la corrección de los tipos en la tabla de símbolos y devuelve \texttt{null}.
	\item \textbf{esTipoTextOrNum}: Devuelve si el tipo pasado es un \textit{string}, o un número.
	\item \textbf{crearPolitipo}: Crea un objeto \texttt{Attrs} del tipo que se le pasa por parámetro, utilizando también un entero contador que lo representa unívocamente. Esta representación es la que permite diferenciar los tipos de politipo (por ejemplo, 'a de 'b).
	\item \textbf{obtenerTipo}: Busca primero en la tabla de símbolos local (si existe) y luego en la global el tipo del identificador pasado por parámetro. En caso de que el identificador no esté declarado, devuelve \texttt{null}.
	\item \textbf{esTipoSimple}: indica si el atributo pasado por parámetro es un \textit{string}, un número real o entero, o un valor booleano.
\end{itemize}


\chapter{Batería de tests}
Los tests fueron divididos en tres partes: válidos, inválidos, y los que presentan bugs que no pudieron arreglarse por cuestiones de tiempo.

\section{Known bugs}
Existen casos en los que el intérprete no se comporta exactamente como lo haría el de ML original; éste tipo de caso no pudo ser controlado por falta de tiempo, y es reportado en esta sección:

\begin{lstlisting}
MLOR> fun f x = x [x];
val f: ('a list -> 'b) -> 'b = fn;
\end{lstlisting}
En este caso, el intérprete original de ML indica un conflicto: "`Circular type results from unifying 'a and 'a list -> 'b"'. Sin embargo, en MLOR no realizamos detección de aplicaciones circulares en funciones, por lo que este caso da una respuesta errónea.

Por otro lado, MLOR puede presentar algunos problemas en las aplicaciones parciales de funciones con parámetros politipo a los que se les instancia un tipo estructurado complejo. Por ejemplo:

\begin{lstlisting}
MLOR> fun h x y = (x,y);
val h: 'a -> 'b -> ('a * 'b) = fn;
MLOR> fun g x = h (x, [1,2,3,4]);
val g: ('a * int list) -> 'b -> ('c * 'b) = fn;
\end{lstlisting}

El tipo de la función g debería ser: 'a -> 'b -> (('a * int list) * 'b) = fn
